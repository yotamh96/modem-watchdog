# Phase 05.4 — dms-revision-parser-hotfix (retroactive)

## Why this phase exists

Phase 05.3 unblocked the libqmi-version probe and the bench Jetson advanced
to the **second** of three Phase 5 X-03 preflight probes: the EM modem's
firmware revision via `qmicli --dms-get-revision`. That probe then failed:

```
unknown fleet triple: failed to compute local fleet triple:
dms_get_revision returned QmiError: reason=unexpected_output
detail='no revisions block in stdout'
```

Captured stdout on the bench (qmicli -p -d /dev/cdc-wdm0 --dms-get-revision):

```
[/dev/cdc-wdm0] Device revision retrieved:
        Revision: 'SWI9X50C_01.14.03.00 b06bd3 jenkins 2020/09/23 10:53:35'
```

## Root cause

`src/spark_modem/qmi/parsers/get_revision.py:25` hardcoded the response
header substring as `"Device revisions retrieved"` (plural). The bench
Jetson's libqmi 1.30.4 / SWI9X50C combination emits **singular** `"Device
revision retrieved"` because the output contains only a `Revision:` line
with no `Boot code:` follow-up — libqmi adapts the header text to the
field count.

The pre-existing fixtures (`get_revision/1.30/standard.txt` and
`get_revision/1.32/standard.txt`) both contain a `Boot code:` line and so
both emit the plural header, masking the singular form from CI.

## Scope

- Replace the substring check in `parse_get_revision` with a regex that
  matches either `Device revisions retrieved` or `Device revision retrieved`.
- Add a `tests/fixtures/qmicli/get_revision/1.30/jetpack-singular.txt`
  fixture captured verbatim from the bench Jetson (SWI9X50C modem,
  singular header, single Revision line, no Boot code line).
- Add a `test_parser_accepts_singular_revision_header_jetpack` regression
  test asserting the new fixture parses to the expected revision string.

## Out of scope

- The fact that the bench Jetson is running SWI9X50C (Sierra MC7455 /
  EM7565 family) instead of the SWI9X30C (EM7421) declared in CLAUDE.md.
  The daemon parser doesn't gate on the firmware family; the **known-fleet
  allow-list** in `/etc/spark-modem-watchdog/known-fleet/` does (Phase 5
  X-03). If the bench's triple `(SWI9X50C_01.14.03.00, <zao_sdk>, 1.30.4)`
  is not in the allow-list, the daemon will refuse to start for a
  different structured reason — that's an operator-supplied fleet-fixture
  concern, not a parser concern. The hardware-target discrepancy with
  CLAUDE.md is worth investigating (is this bench Jetson representative
  of the fleet?) but it's a deployment question, not a 05.4 question.
- The Phase 5 X-02 `ctl capture-fleet-fixture` CLI verb's flag-name
  mismatch (it takes `--out`, not `--output`). Cosmetic issue; tracked
  as a follow-up note in this file.

## Discovery context

Caught on the bench Jetson at deploy time (2026-05-12 ~11:20 UTC) after
the Phase 05.3 deploy of commit `4d99a0d` proved the daemon could now
clear the libqmi-version probe. Same one-by-one probe-walking pattern as
05.3: each preflight stage exposes its next gap. The Phase 5 fixture
corpus was assembled from upstream Debian's libqmi + SWI9X30C
documentation; the bench Jetson is the first real-hardware surface to
exercise the parser against an SWI9X50C / no-Boot-code combination.

After 05.4, the third and final probe in `compute_fleet_triple` (the Zao
SDK version detector against `/var/log/zao/zao.log`) is next. That may or
may not work depending on whether Zao has been logging since boot. If it
fails, Phase 05.5 follows.
