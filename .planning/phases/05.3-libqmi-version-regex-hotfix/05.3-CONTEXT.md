# Phase 05.3 — libqmi-version-regex-hotfix (retroactive)

## Why this phase exists

Phase 05.2 retired the `/tmp/spark-modem-cli` EROFS bug and the daemon
advanced past `_ensure_dirs` into the Phase 5 X-03 preflight check on the
bench Jetson. The preflight then rejected the box with:

```
unknown fleet triple: failed to compute local fleet triple:
qmicli --version stdout did not match libqmi-glib regex:
b'qmicli 1.30.4\nCopyright (C) 2012-2021 Aleksander Morgado\n...'
```

Exit code 78/CONFIG (EX_CONFIG) — the structured "configuration invalid"
path in `_production_main`. The daemon refused to start, and systemd
auto-restarted in a loop.

## Root cause

`src/spark_modem/qmi/version.py:30` matched only the optional
`Compiled with libqmi-glib X.Y.Z` footer line in `qmicli --version` output.
On JetPack 5.1.5 / Ubuntu 20.04 the bundled `libqmi 1.30.4` build does NOT
include that footer line; only the first-line `qmicli 1.30.4` banner is
emitted. The existing test fixtures (`1.30/standard.txt` for libqmi 1.30.6,
`1.32/standard.txt` for libqmi 1.32.0) both happen to include the footer
because they were captured from upstream Debian builds — the JetPack build
shape was never represented in the test corpus.

`qmicli` and `libqmi-glib` are versioned in lockstep (qmicli is a frontend
that statically/dynamically links libqmi-glib from the same source tree), so
the version reported on either line is byte-identical.

## Scope

- Update `_LIBQMI_VERSION_RE` to accept either `qmicli X.Y.Z` OR
  `libqmi-glib X.Y.Z`.
- Add a `tests/fixtures/qmicli/version/1.30/jetpack-1.30.4.txt` fixture
  capturing the bench Jetson's exact stdout (no libqmi-glib footer).
- Add a unit test `test_detect_libqmi_version_parses_jetpack_qmicli_only_format`
  asserting `"1.30.4"` is parsed from the new fixture.

## Out of scope

- Changing how `compute_fleet_triple` orchestrates the three probes (Phase 5
  X-03 contract). The triple itself remains `(em7421_firmware, zao_sdk, libqmi)`.
- The `known-fleet/` directory contents (the operator-supplied allowed-triple
  set). The bench Jetson's triple `(SWI9X30C_..., 2.1.0, 1.30.4)` may or may
  not match an existing known-fleet entry — that is a deployment-time
  question, not a regex-parser question.
- Adding `--skip-preflight` as a permanent daemon flag. It already exists for
  testing; operators should not run with it in production because the X-03
  triple check is a real safety gate.

## Discovery context

Caught on the bench Jetson at deploy time (2026-05-12 ~10:25 UTC) after the
Phase 05.2 deploy of commit `e49dc7b` proved the daemon could now reach the
preflight step at all. The failure mode shifted from `1/FAILURE` EROFS to
`78/CONFIG` structured rejection — the latter is the "the daemon is alive
enough to refuse cleanly" signal. CI's V-02 install gate does not invoke the
daemon binary (only `ctl config-check`), so the regex mismatch would not
have been caught in CI. Real-hardware deploy is the right surface for it,
which is precisely what Phase 5 X-03 was designed to catch — the X-03 gate
is doing its job, just with an over-narrow regex.
