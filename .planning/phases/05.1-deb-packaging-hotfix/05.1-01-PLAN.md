<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build-time runner → built `.deb` artifact | a malicious source tree or upstream uv could poison the bundled site-packages |
| dpkg postinst (root) → /opt/spark-modem-watchdog/python/ | runs with root privilege; venv layout is what daemon `import`s under root |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.1-01 | T (tampering) | `uv pip install . --no-build-isolation` step in debian/rules | mitigate | `--no-deps` enforces "only the local source"; runtime deps are separately locked at step 3 via `--require-hashes`; the local source comes from the GitHub-checked-out tree (no network fetch for spark_modem itself). Plan 05's V-02 CI install test re-verifies the resulting `.deb` in a clean container. |
| T-05.1-02 | I (info disclosure) | spark_modem package shipped in /opt/.../site-packages/ | accept | spark_modem is the application code itself; shipping it in site-packages is by design. No secrets baked in (HMAC discipline lives in Plan 02). |
| T-05.1-03 | D (DoS / availability) | NFR-51 size budget (≤40 MiB .deb) regressed by spark_modem install | mitigate | spark_modem source tree is ~few hundred KB Python; `--no-deps` prevents resolver expansion; the existing `verify-deb-meta` step in `.github/workflows/build-deb.yml` lines 35-48 asserts `test "$SIZE_MIB" -le 40` and fails the build if exceeded. No new mitigation needed; existing CI gate covers it. |
</threat_model>
