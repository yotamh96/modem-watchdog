# Phase 05.1: deb-packaging-hotfix - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Make `dpkg -i spark-modem-watchdog_2.0.*_arm64.deb` followed by
`systemctl start spark-modem-watchdog.service` reach `active (running)` with
`sd_notify READY=1` on a bench Jetson (JetPack 5.1.5 / Ubuntu 20.04 /
systemd 245 / aarch64). Three known bugs block this today; fix them, ALSO
land a regression gate so the same class of bug cannot recur silently.

**The three bugs (confirmed in tree at discuss time):**

1. **`spark_modem` not on `sys.path`** — `debian/spark-modem-watchdog.install:6`
   installs source to `/opt/spark-modem-watchdog/lib/spark_modem/`, but the
   bundled venv only sees `/opt/spark-modem-watchdog/python/lib/python3.12/
   site-packages/`. The B-03 postinst smoke test only imports the 10 runtime
   libs, never `spark_modem` itself — which is why this slipped past Phase 1
   CI.
2. **No daemon entry point** — `pyproject.toml [project.scripts]` declares
   only `spark-modem`. The systemd unit (`debian/spark-modem-watchdog.service:23`)
   references `/opt/spark-modem-watchdog/bin/spark-modem-watchdog`, which
   nothing creates → `systemd` raises `203/EXEC`. The unit file even carries
   a "Phase 4 follow-up: verify wrapper exists; current .deb scaffold may
   need an additional ExecStart shim" comment admitting the gap.
3. **`LoadCredential=` on systemd 245** — `debian/spark-modem-watchdog.service:100`
   uses `LoadCredential=` (introduced in systemd 247). Ubuntu 20.04 ships
   systemd 245. Behavior at parse time (hard-fail vs silent-ignore) is
   unverified and locks the fallback design — see decision **L-04**.

**Out of scope for 05.1:**

- requirements.lock refresh (the 10 runtime libs do not change here)
- Any broader src/ work beyond the HMAC-secret fallback path
- ROADMAP Success Criteria block for 05.1 (we use an EXIT bar pattern,
  same as Phase 4 "first nightly HIL run")
- `.deb` pipeline review beyond the immediate bug surfaces (deferred)

**Phase 05.1 EXIT bar:** bench Jetson runs the new `.deb`, daemon reaches
`active (running)` + `sd_notify READY=1`, structured `EXIT-CHECKLIST.md`
committed by the operator. Unblocks Plan 05-08 Task 2 bench soak window.

</domain>

<decisions>
## Implementation Decisions

### Scope discipline (D-*)

- **D-01:** Hotfix shape is **minimum + regression gate**. Fix the 3 bugs;
  also extend the B-03 smoke to import `spark_modem.daemon.main` +
  `spark_modem.cli.main`; also add a CI install test so this class of bug
  cannot recur. Bug class permanently retired by exit.
- **D-02:** **pyproject.toml only** — no requirements.lock churn. Adding
  `spark-modem-watchdog` to `[project.scripts]` is the sole pyproject edit.
  `uv pip install . --no-deps` ships spark_modem itself; the 10 runtime
  libs stay pinned exactly where Phase 1 placed them.
- **D-03:** **EXIT bar pattern**, not a full ROADMAP SC block. EXIT =
  bench Jetson green per **V-03** below. Mirrors Phase 4's "first nightly
  HIL run" exit-bar idiom.
- **D-04:** **Glue + tiny daemon-side hook**. Most diff lives under
  `debian/*` + `scripts/*` + `pyproject.toml`. Exactly one `src/` change:
  the HMAC-secret fallback path in `config/settings.py` (see L-02).

### Install + entry-point + bin/ layout (I-*)

- **I-01:** **`uv pip install . --no-deps --no-build-isolation`** runs in
  `override_dh_auto_install` after the 10 runtime libs land. This drops
  `spark_modem` into the bundled venv's site-packages as a properly
  installed package; `[project.scripts]` console-scripts auto-materialize
  at `/opt/spark-modem-watchdog/python/bin/`. The current `src/spark_modem
  /opt/spark-modem-watchdog/lib/` line in `debian/spark-modem-watchdog.install`
  is **removed** — no duplicate source tree shipped.
- **I-02:** Daemon entry point lives in `pyproject.toml [project.scripts]`:
  ```
  spark-modem-watchdog = "spark_modem.daemon.main:_sync_main"
  ```
  Symmetric with the existing `spark-modem = "spark_modem.cli.main:main"`.
- **I-03:** `/opt/spark-modem-watchdog/bin/` is **dropped entirely**. The
  systemd unit is edited so all `ExecStartPre`/`ExecStart` lines reference
  `/opt/spark-modem-watchdog/python/bin/spark-modem` and
  `/opt/spark-modem-watchdog/python/bin/spark-modem-watchdog` directly. One
  fewer abstraction layer; cleanest pairing with the pip-install path.
- **I-04:** The sync wrapper around the existing `async def main()` lives
  **inline** in `src/spark_modem/daemon/main.py`:
  ```python
  def _sync_main(argv: list[str] | None = None) -> int:
      return asyncio.run(main(argv))
  ```
  Co-located with the async `main()`; mypy-strict clean; no new module.
- **I-05:** `uv pip install .` requires `setuptools` to be present in the
  bundled venv at install time (the build backend declared in
  `pyproject.toml`). It is already installed transitively by the `python
  -m ensurepip --upgrade` + `pip install uv` steps in `debian/rules`. The
  uninstall sweep at the end of `override_dh_auto_install` still strips
  setuptools after `uv pip install .` completes — verified by NFR-51 size
  budget (≤40 MiB). Planning must confirm setuptools survives until AFTER
  the spark_modem install runs, not before.

### LoadCredential fallback + HMAC secret lifecycle (L-*)

- **L-01:** **`LoadCredential=` stays in the unit file.** The directive is
  the forward-compatible path for Ubuntu 22.04+ / systemd 247+ boxes
  (Phase 6+ may upgrade Jetson hosts). On systemd 245 boxes a code-side
  fallback kicks in — see L-02.
- **L-02:** Daemon-side fallback in `src/spark_modem/config/settings.py`:
  if `CREDENTIALS_DIRECTORY` env var is set (systemd 247+ path), read the
  HMAC secret from `${CREDENTIALS_DIRECTORY}/spark-modem-watchdog.hmac-secret`.
  Else fall back to reading **directly** from
  `/etc/spark-modem-watchdog/hmac-secret`. The fallback path matches the
  LoadCredential source path so a single file on disk serves both worlds.
- **L-03:** HMAC secret file lifecycle = **operator pre-start, placeholder
  rejected**. Postinst writes
  `/etc/spark-modem-watchdog/hmac-secret` as a 0600 root:root file
  containing the literal sentinel `REPLACE_THIS_BEFORE_FIRST_START_SEE_RUNBOOK\n`.
  Operator runs `head -c 32 /dev/urandom | base64 | install -m 0600
  -o root -g root /dev/stdin /etc/spark-modem-watchdog/hmac-secret` before
  first start. Matches ADR-0011 "never baked in".
- **L-04:** **Plan must verify experimentally** whether systemd 245
  hard-fails parsing `LoadCredential=` or silently ignores. Task: spin a
  clean Ubuntu 20.04 arm64 container (or use the existing Docker CI
  install harness from V-02), paste the .service file with `LoadCredential=`
  present, run `systemd-analyze verify`. Three branches:
  - **Silent-ignore:** L-01 + L-02 are sufficient; no further unit changes.
  - **Hard-fail:** add a postinst-managed drop-in override at
    `/etc/systemd/system/spark-modem-watchdog.service.d/10-systemd245-no-loadcredential.conf`
    that empties `LoadCredential=` when systemd version is detected <247.
    Canonical unit file in `debian/` stays pristine.
  - **Warning-with-degraded-state:** treat as silent-ignore; document the
    journalctl warning operators will see on 20.04 boxes in RUNBOOK.
- **L-05:** `spark-modem ctl config-check` (the existing ExecStartPre verb
  from Plan 03-08) is **extended** to verify the HMAC secret file:
  (a) exists at the path settings resolved (CREDENTIALS_DIRECTORY-derived
  or fallback); (b) is **not** the placeholder sentinel; (c) is 0600,
  owner root, group root; (d) is non-empty. Failure surfaces as a clear
  structured error before the main daemon process ever boots — same shape
  as the existing config-check failures.

### Validation gate (V-*)

- **V-01:** **B-03 smoke test extended** — `scripts/postinst_smoke_test.sh`
  imports the 10 runtime libs (today) plus `spark_modem.daemon.main` and
  `spark_modem.cli.main`. Two-line addition. Catches the bug class at
  install time (postinst) AND on every start (ExecStartPre). Permanently
  retires bug #1.
- **V-02:** **New dockerized CI install job** on the self-hosted aarch64
  runner. Steps: build the `.deb` from the same workflow, spin
  `--platform linux/arm64 ubuntu:20.04`, `apt-get install -y
  ./packaging/output/spark-modem-watchdog_*.deb`, assert: postinst smoke
  green, `/opt/spark-modem-watchdog/python/bin/spark-modem-watchdog`
  exists and is executable, `/opt/spark-modem-watchdog/python/bin/spark-modem`
  exists, `/etc/spark-modem-watchdog/hmac-secret` exists with 0600 mode
  and placeholder content, `systemd-analyze verify
  /lib/systemd/system/spark-modem-watchdog.service` exits 0 (catches L-04
  hard-fail case). Linux-only; runs on every push to main + every PR.
- **V-03:** **Bench Jetson EXIT bar** — operator-facing structured
  checklist committed at
  `.planning/phases/05.1-deb-packaging-hotfix/EXIT-CHECKLIST.md`. Mirrors
  Phase 5's SIGNOFF.md template shape. Steps (each pass/fail):
  1. `.deb` built from the merged hotfix branch
  2. `scp` to bench Jetson, `dpkg -i` returns 0
  3. Operator provisions `/etc/spark-modem-watchdog/hmac-secret`
  4. `systemctl start spark-modem-watchdog.service` returns 0
  5. `systemctl is-active spark-modem-watchdog.service` reports `active`
  6. `journalctl -u spark-modem-watchdog.service` shows `Started ...` +
     no ERROR/CRITICAL lines
  7. `/run/spark-modem-watchdog/lock` present + owned by root
  8. `/run/spark-modem-watchdog/metrics.sock` present, scrape returns
     valid Prometheus text
  9. Daemon reaches Healthy on all 4 modems within 60s (NFR-13)
  Phase 05.1 EXIT = committed checklist with every box ticked.
- **V-04:** **Unit-file audit extended** — Plan 03-08's
  `tests/integration/test_unit_file_audit.py` gains 3 new assertions:
  (a) every ExecStart/ExecStartPre path appears in `pyproject.toml
  [project.scripts]` OR in `debian/spark-modem-watchdog.install` OR is a
  systemd-managed runtime path; (b) `LoadCredential=` directive presence
  is matched against a documented allow-list of systemd versions
  (≥247 = OK, <247 = requires fallback documented in code); (c)
  console-script entry points named in `[project.scripts]` are
  asserted-importable from this test process (`importlib.metadata.entry_points()`).
  Cross-platform; runs on every Windows dev-host pytest invocation. Catches
  future drift of unit ↔ pyproject ↔ install layout.

### Claude's Discretion

- The **exact** Docker image tag for V-02 (`ubuntu:20.04` vs
  `arm64v8/ubuntu:20.04` vs a pinned-by-digest variant). Lowest-friction
  choice that runs reliably on the existing self-hosted aarch64 runner.
- The **changelog versioning bump** shape — `2.0.0 → 2.0.1`
  (PEP-440 patch) vs `2.0.0+hotfix.1` (PEP-440 local-version). Plan picks
  the cleanest path; debian/changelog entry shape follows.
- Whether `spark-modem-watchdog --help` should print non-trivial usage
  (today it just falls through to argparse defaults). If it surfaces a
  `--version` flag in passing, V-02 can scrape it — but the hotfix
  doesn't NEED `--version` to validate.
- The **exact placeholder sentinel string** (L-03). Long-enough to be
  obviously non-secret; collision with real base64 vanishingly unlikely.
- The **exact shape** of the L-04 verification step inside the plan —
  whether it lands as a CI assertion (V-02 verifies it) or a one-shot
  research task that updates the plan branch decision. The CI assertion
  is the cleanest forcing function.
- Whether the daemon's `_sync_main` should propagate `KeyboardInterrupt`
  cleanly (it inherits from `asyncio.run` semantics; fine for systemd
  Type=notify which sends SIGTERM, never SIGINT).

</decisions>

<specifics>
## Specific Ideas

- The bench Jetson EXIT-CHECKLIST.md is single-operator; the on-site
  engineer who runs Phase 5 Plan 05-08 bench soak also runs this. No new
  human is in the loop. Step 3 (operator provisions HMAC secret) is the
  single non-automatable step.
- `uv pip install . --no-deps --no-build-isolation` is the canonical
  invocation — `--no-build-isolation` reuses the setuptools already
  installed in the bundled venv (cheap, deterministic) rather than
  letting uv spin a fresh build venv (expensive, network-touching,
  defeats reproducibility).
- The Docker CI install test in V-02 is the natural place to also run
  `systemd-analyze verify` and capture the L-04 verdict (silent-ignore
  vs hard-fail). If the verdict is hard-fail, V-02 surfaces it as a CI
  failure and the plan adds the drop-in override path. The L-04
  verification + the V-02 CI test collapse into one task.
- The current `pyproject.toml` `[tool.setuptools.packages.find]` is
  already correct (`where = ["src"]`, `include = ["spark_modem*"]`) —
  `uv pip install .` will pick up the package without further config.
- The HMAC-secret placeholder file is mode 0600 root:root; the directory
  `/etc/spark-modem-watchdog/` is already created by Phase 1's postinst
  via `install -d` (configurable, default 0755).
- Plan 03-08's existing config-check ExecStartPre already runs as root
  (the daemon runs as root per NFR-30); reading a 0600 root file in L-05
  is straightforward.
