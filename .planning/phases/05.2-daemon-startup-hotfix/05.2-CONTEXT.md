# Phase 05.2 — daemon-startup-hotfix (retroactive)

## Why this phase exists

The Phase 05.1 hotfix successfully unblocked the .deb packaging pipeline: a
bench Jetson install of `spark-modem-watchdog_2.0.0-0.gite49dc7b-1_arm64.deb`
now passes both `ExecStartPre` gates (postinst smoke + `ctl config-check`) and
reaches the main `ExecStart`. At that point the deploy surfaced a *separate*
bug — not in packaging, not in the unit file, but in the daemon itself:

```
File "/opt/.../site-packages/spark_modem/daemon/main.py", line 207, in _production_main
    _ensure_dirs(state_root_path, run_dir, Path(settings.events_log_path).parent)
OSError: [Errno 30] Read-only file system: '/tmp/spark-modem-cli'
```

`/tmp/spark-modem-cli` is not a production daemon path. It is the CLI
laptop-sandbox path declared by `spark_modem.cli.clients.build_default_settings()`
for `recovery --diag-fixture=...` runs on a developer laptop.
`daemon/main.py:_production_main` was inadvertently importing and calling
`build_default_settings()` instead of constructing `Settings()` directly,
forcing the daemon to try and mkdir into a tree under `/tmp/` — which is
correctly read-only inside the systemd unit's hardened namespace
(`ProtectSystem=strict`).

## Scope

Single-task hotfix:

- Swap `build_default_settings()` for `Settings()` at `daemon/main.py:192` so
  the daemon uses production defaults (`/var/lib/`, `/run/`, `/var/log/`)
  with normal pydantic-settings env-var overrides.
- Leave `_laptop_main` (the `--laptop` integration-test wiring) unchanged —
  its use of `build_default_settings()` at line 119 is correct for that
  fixture-driven path.

## Out of scope

- Adding more granular settings tests beyond the type-check + lint that
  already cover `daemon/main.py`. Regression of this exact bug class
  (wrong settings factory in production main) is best-caught by an
  integration test that boots the daemon under a temp HOME without
  `/tmp/` write access and asserts no `OSError` on `_ensure_dirs` — but
  that is gardening, not hotfix scope. Leave for Phase 6 hardening if
  it ever recurs.
- The `/tmp/spark-modem-cli` constants themselves. They are correct for
  the CLI laptop sandbox. Don't change them.

## Discovery context

Caught on the bench Jetson at deploy time (2026-05-12) after the Phase 05.1
chain of fixes (78d1359 → 56a6ab1 → 7d45b14 → 10cec6d → e49dc7b) walked the
.deb through `dpkg -i` cleanly. The Phase 05.1 V-02 CI gate exercises
`spark-modem ctl config-check` end-to-end but does NOT exercise the
`spark-modem-watchdog` daemon binary — the daemon's ExecStart would have
needed a hardened-namespace environment to surface this bug class, which the
container test step does not provide. Worth considering for a Phase 6
"daemon-namespace dry-start" CI gate.
