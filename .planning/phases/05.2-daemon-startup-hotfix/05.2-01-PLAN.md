# Plan 05.2-01 — Use Settings() directly in `_production_main`

## Objective

Replace the call to `build_default_settings()` inside `daemon/main.py:_production_main`
with `Settings()`. This swaps the CLI laptop-sandbox factory (which hardcodes
every path under `/tmp/spark-modem-cli/`) for the production-defaults factory
(`/var/lib/spark-modem-watchdog`, `/run/spark-modem-watchdog`,
`/var/log/spark-modem-watchdog/...`), with normal pydantic-settings
env-var override semantics on top.

The daemon's `_laptop_main` function (the `--laptop` integration-test path)
keeps using `build_default_settings()` — that is correct for fixture-driven
laptop runs.

## Must-haves

### Truths
- `daemon/main.py:_production_main` constructs `Settings()` directly, no
  longer calling `build_default_settings()`.
- `daemon/main.py` imports `Settings` from `spark_modem.config.settings`.
- `daemon/main.py:_laptop_main` continues to call `build_default_settings()`
  (unchanged).
- `mypy --strict src/spark_modem/daemon/main.py` is green.
- `ruff check src/spark_modem/daemon/main.py` is green.

### Artifacts
- path: src/spark_modem/daemon/main.py
  provides: production Settings construction in `_production_main`
  contains: `settings = Settings()` inside the Step 2 try-block
- path: src/spark_modem/daemon/main.py
  provides: production-source Settings import
  contains: `from spark_modem.config.settings import Settings`

### Key links
- from: `daemon/main.py:_production_main` Step 2
  to: `spark_modem.config.settings.Settings` class
  via: direct construction, no factory indirection
  pattern: `settings = Settings()`

## Tasks

### Task 1 — Add the production Settings import

Open `src/spark_modem/daemon/main.py`. After the existing
`from spark_modem.cli.clients import (...)` block (which keeps
`build_default_settings` for `_laptop_main`), add a new import line:

```python
from spark_modem.config.settings import Settings
```

### Task 2 — Swap the factory call inside `_production_main`

Inside `_production_main`, at the Step 2 `try:` block, replace:

```python
settings = build_default_settings()
```

with:

```python
settings = Settings()
```

Add an inline comment immediately above the try-block documenting the
2026-05-12 bench Jetson deploy failure and the reason for the swap (see
`05.2-CONTEXT.md`). The comment should mention:
- production defaults come from `config/settings.py`
- pydantic-settings layers env-var overrides on top
- `build_default_settings()` is the CLI laptop sandbox; do not use it here
- the unit's `ProtectSystem=strict` correctly renders `/tmp` read-only

### Task 3 — Verify

Run:
```bash
uv run mypy --strict src/spark_modem/daemon/main.py
uv run ruff check src/spark_modem/daemon/main.py
```

Both must exit green. No other source files change.
