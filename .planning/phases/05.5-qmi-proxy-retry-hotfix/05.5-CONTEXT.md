# Phase 05.5 — qmi-proxy-retry-hotfix (retroactive)

## Why this phase exists

Phase 05.4 broadened `parse_get_revision` to accept the singular header
form the bench Jetson emits — and the parser fix verifiably deployed
(grep on the installed `.py` confirmed `Device revisions? retrieved`).
Yet the daemon continued to fail with the same surface error:

```
unknown fleet triple: failed to compute local fleet triple:
dms_get_revision returned QmiError: reason=unexpected_output
detail='no revisions block in stdout'
```

Manual probes against the four modems revealed the real cause: **qmi-proxy
CID allocation races with Zao**. Three sequential manual runs (after
Phase 05.4 install) showed a different device failing each time:

| Run | wdm0 | wdm1 | wdm2 | wdm3 |
|-----|------|------|------|------|
| 1 | ✓ | ✓ | ✗ Service mismatch (nas) | ✓ |
| 2 | ✓ | ✗ Service mismatch (nas) | ✗ endpoint hangup | ✓ |
| 3 | ✗ Service mismatch (uim) | ✓ | ✓ | ✗ Service mismatch (nas) |

Zao runs continuous NAS / UIM probes on all four lines through the same
`qmi-proxy` we share with `--device-open-proxy`. When our daemon's CID
allocation arrives mid-flight against a Zao request, qmi-proxy returns
the wrong service handle and qmicli aborts with "Service mismatch
(requested 'dms', got 'nas'|'uim')" or "endpoint hangup". stdout in
those cases is empty (the error goes to stderr), so the parser
correctly reports `UNEXPECTED_OUTPUT` — but the actual cause is
upstream contention, not a parser issue.

## Root cause

`compute_fleet_triple` (`src/spark_modem/qmi/version.py:140`) calls
`wrapper.dms_get_revision()` exactly once and passes `cp.stdout`
straight to the parser, **ignoring `cp.exit_code` and `cp.stderr`**.
Any qmicli-level failure surfaces as a parser-level UNEXPECTED_OUTPUT,
hiding the real diagnostic.

## Scope

- Retry the firmware probe up to 3 times with 0.5s backoff between
  attempts.
- Check `cp.exit_code` before invoking the parser; on non-zero exit,
  capture `cp.stderr` (truncated) and retry.
- On final failure, raise `QmiVersionDetectionFailed` with a message
  that names BOTH the attempt count AND the underlying cause (either
  `qmicli exit_code=N stderr=...` or `parser returned QmiError: ...`).
- Add a unit test that exercises a transient-failure-then-success
  sequence and verifies all three attempts were used.
- Add a unit test that exercises all-attempts-fail and verifies the
  final error includes the qmicli stderr.
- Update the existing `test_compute_fleet_triple_firmware_qmierror_raises`
  match string to reflect the new error format (the test's semantic
  intent is preserved — malformed stdout still raises — only the
  wording of the raised message changes).

## Out of scope

- Retry logic for the libqmi version probe (`detect_libqmi_version`).
  That call uses bare `qmicli --version` without `-p`, doesn't touch
  qmi-proxy, doesn't allocate CIDs, and the bench Jetson deploy walk
  showed it succeeding on the first attempt every time.
- Retry logic for `cycle_driver` diag probes. The same race almost
  certainly affects them too, but the daemon doesn't get to cycle
  steady-state until preflight passes — fix preflight first, observe
  the next failure layer in production, then iterate.
- Removing `qmi-proxy` reliance. The proxy is mandatory because Zao
  needs simultaneous access to the same modems. The architectural
  alternative (split CID-pool reservation, or a fairness scheduler in
  the proxy) is a separate, much larger project.
- The Sierra firmware family mismatch with CLAUDE.md (SWI9X50C on
  bench vs SWI9X30C / EM7421 in spec). Still benign, still a flagged
  separate operational concern.

## Discovery context

Caught at deploy time on 2026-05-12 (~12:20 UTC) when Phase 05.4's
parser fix deployed cleanly but the daemon kept producing the same
error. Manual probe runs showed the per-device flakiness pattern,
which the daemon was hitting on every preflight start.

CI cannot reproduce this because the V-02 install gate runs in a
clean Ubuntu 20.04 container with no Zao process competing for the
proxy. The bench Jetson is the right surface for it — the failure
mode genuinely depends on a Zao process running concurrently.

## Note on retry-budget calibration

Three attempts with 0.5s spacing is empirical. The bench data shows
~25% per-call failure rate when Zao is active, which gives a single-
attempt success of 75%, three-attempt cumulative success of
1 - 0.25³ = 98.4%. That clears the preflight gate reliably without
spending more than ~1s of wallclock on the retry budget. If
real-world failure rates prove higher, bump `_FIRMWARE_PROBE_ATTEMPTS`.
