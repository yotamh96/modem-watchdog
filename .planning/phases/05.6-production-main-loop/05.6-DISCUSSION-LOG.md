## Cycle interval + warmup

### Q: What production cycle interval should the daemon use?

| Option | Description | Selected |
|--------|-------------|----------|
| 30s | CycleScheduler current default; Phase 2/3 fixture cadence; faster MTTR. WatchdogSec=90 = 3× cycle. Risk: more qmi-proxy contention with Zao. | |
| 60s | Compromise; halves cycle-time qmi-proxy interaction rate. WatchdogSec→180. M2 MTTR targets still reachable on first-bad-cycle. | ✓ |
| 120s | v1 inherited cadence; minimum contention. WatchdogSec=360. M2 60s-SIM MTTR target hard to hit. | |

**User's choice:** 60s
**Consequence captured:** WatchdogSec must bump 90 → 180 in `debian/spark-modem-watchdog.service`.

### Q: Where should the cycle interval live?

| Option | Description | Selected |
|--------|-------------|----------|
| Settings field RELOAD_DATA | New `cycle_interval_seconds` field; SIGHUP can retune live during shadow/canary. Mirrors `ladder_min_interval_seconds` / `multi_modem_threshold_fraction`. | ✓ |
| Hardcoded in `_production_main` | Literal at CycleScheduler construction site. Loses operational tunability. | |
| Settings field RELOAD_RESTART | Settings-configurable but topology-tagged. Loses "retune live under shadow" use case. | |

**User's choice:** Settings field RELOAD_DATA

### Q: How should the daemon behave during the first cycle if qmi-proxy contention makes probes slow?

| Option | Description | Selected |
|--------|-------------|----------|
| Strict + STATUS progress | READY=1 only after cycle 0 (PITFALLS §4.1); per-modem STATUS during warmup for operator visibility. If cycle 0 > 60s, systemd correctly fails Type=notify. | ✓ |
| Strict only | SPEC verbatim: no intermediate STATUS during warmup. Operator sees nothing until cycle completes or 60s timeout. | |
| Early READY after preflight + lock | READY=1 before cycle 0. Violates "meaningful readiness" — daemon could report Active while cycle 0 hung. | |

**User's choice:** Strict + STATUS progress

### Q: What STATUS= cadence and content per cycle (after warmup)?

| Option | Description | Selected |
|--------|-------------|----------|
| Per-cycle terse | `cycle={N} healthy={K}/4 actions={M} drift={X}s` — Phase 3 L-01 pinned format. One update per cycle at cycle-end. | ✓ |
| Per-cycle + per-action | Plus a STATUS during action-execute. More visibility but doubles sd_notify traffic. | |
| Per-cycle verbose | Include last-action and last-error. Pushes status.json territory into systemctl status output. | |

**User's choice:** Per-cycle terse

## Integration-test strategy

### Q: What test surface(s) should pin the main-loop regression?

| Option | Description | Selected |
|--------|-------------|----------|
| In-process main() + fake sd_notify | `await main(['--skip-preflight'])`; monkeypatch SdNotifyLifecycle or NOTIFY_SOCKET = tmp abstract socket. Cheapest CI; runs on Linux dev + aarch64. Would catch placeholder. | ✓ |
| Subprocess main + real NOTIFY_SOCKET | Spawn `python -m spark_modem.daemon.main` with NOTIFY_SOCKET = tmp socket; recv READY=1. Most faithful; slower; Linux CI only. | |
| Both layers | In-process for fast PR gate + subprocess for nightly. Max coverage; 2× maintenance. | |
| testcontainers / systemd-in-docker | Privileged container, install .deb, systemctl start. Most realistic; brittle, slow. Bench Jetson human-verify covers it. | |

**User's choice:** In-process main() + fake sd_notify

### Q: What should the test inject for the four event producers?

| Option | Description | Selected |
|--------|-------------|----------|
| All Fake* | FakeUdevMonitor / FakeAsyncIPRoute / FakeAsyncinotify / FakeKmsgReader (Phase 3). Pins the WIRING; real bindings covered by producer plan tests. | ✓ |
| Real producers on Linux CI | Real pyudev/pyroute2/asyncinotify/kmsg. Catches real-binding bugs; risk: flakiness without USB modems / kmsg permission. | |
| Mix | Fakes for udev/rtnetlink/asyncinotify, real /dev/kmsg. More realistic without quad-binding fragility. | |

**User's choice:** All Fake*

### Q: Where does the new test live?

| Option | Description | Selected |

### Q: What does the test assert beyond READY=1?

| Option | Description | Selected |
|--------|-------------|----------|
| READY + status.json + 1 cycle metric | READY=1 fires; status.json has 4 modems; `actions_total` or `cycle_duration_seconds` incremented. Catches wiring AND cycle execution. | ✓ |
| READY + WATCHDOG + SIGTERM round-trip | Plus WATCHDOG=1 fires ≥1× and DaemonStopped event ≤5s on SIGTERM. Closes loop on cycle-end placement + graceful shutdown. | |
| READY only | Just READY=1. Catches return-0 placeholder; misses cycle-end placement and SIGTERM regressions. | |

**User's choice:** READY + status.json + 1 cycle metric

## qmi-proxy retry at cycle time

### Q: How should cycle-time qmi-proxy contention be handled in this phase?

| Option | Description | Selected |
|--------|-------------|----------|
| Defer — observe in production | SPEC's call. Ship unchanged QmiWrapper; cycle-time failures surface as Issue(category=qmi). Iterate if logs show the race firing. | ✓ |
| Preemptive retry at QmiWrapper layer | Wrap all 11 qmicli methods with Phase 05.5 pattern. Hides intermittent failures that inform diagnosis. | |
| Preemptive retry, but in subproc/runner | Generalize at subprocess layer. Couples retry policy to runner (supposed to be pure transport). | |
| Defer + cycle-time observability hook | No retry; add `qmicli_failures_total{intent,reason}` metric for shadow signal. | |

**User's choice:** Defer — observe in production

### Q: If the race surfaces in shadow / canary, what trigger opens a follow-up phase?

| Option | Description | Selected |
|--------|-------------|----------|
| ≥2 cycle-time qmicli failures per modem per hour | Concrete operability gate, fleet-aggregate. Phase 5/6 already tracks per-cycle error rates. | ✓ |
| Any cycle-time qmicli failure in production | Zero-tolerance. Overreacts to single transient failures. | |
| Cycle exceeded 10s NFR-1 P99 budget due to qmi-proxy | Performance-anchored. Requires `cycle_duration_seconds` reason-attribution; bigger instrumentation lift. | |

**User's choice:** ≥2 cycle-time qmicli failures per modem per hour

## Plan slicing

### Q: How should the phase be sliced into plans?

| Option | Description | Selected |
|--------|-------------|----------|
| Accept SPEC's 5-plan suggestion | 01 skeleton+sd_notify+signals; 02 producers; 03 cycle body; 04 SIGTERM/SIGHUP; 05 integration test. Each independently bench-PASSable. | ✓ |

**User's choice:** Accept SPEC's 5-plan suggestion

### Q: What gating between plans is needed (esp. bench Jetson checkpoints)?

| Option | Description | Selected |
|--------|-------------|----------|

| Bench-PASS only at end | All 5 land, then one final verification. Faster; bigger blast radius mid-chain. | |

**User's choice:** Bench-PASS after 01 and after 05
