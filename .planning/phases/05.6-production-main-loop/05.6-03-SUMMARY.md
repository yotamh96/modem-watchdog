# Phase 05.6 Plan 03: Production Cycle Body Summary

**Replaces plan 05.6-01's stub `_cycle_loop` with the production cycle body. Each cycle invokes `CycleDriver.run_one_cycle(cycle_id)` which writes `status.json` internally; the loop then kicks the systemd watchdog, emits a terse STATUS line, fires READY on cycle 0, and advances the scheduler. The webhook poster and Prometheus UDS server are added as the 5th + 6th TaskGroup tasks (9 tasks total: 4 producers + webhook + prom + cycle + sigterm + sighup). The boot envelope (`DaemonRestart` with the classified prior reason) is enqueued onto the webhook poster BEFORE TaskGroup entry. The HMAC secret is loaded via `Settings.resolve_hmac_secret_path()` with graceful FileNotFoundError handling; the carrier table is loaded via `yaml.safe_load + CarrierTable.model_validate` with graceful degradation on parse failure. The cycle driver, scheduler, store, metrics, carrier table, and webhook poster are constructed once each in the `_production_main` body, between the producer scaffolding and the producer factories.**

## Decisions Made

- **Prometheus UDS server starts BEFORE cycle 0.** Per CONTEXT.md Claude's Discretion: lean toward before; the scrape protocol tolerates empty registries (Prom's `make_wsgi_app(REGISTRY)` serves an empty `# HELP` / `# TYPE` block before any observations land — operators scraping during warmup see an empty body, not ECONNREFUSED). This matches plan 05.6-05's T-04 assertion shape ("at least one of `actions_total` / `cycle_duration_seconds` recorded") which expects the socket bound during cycle 0.

- **ASYNC240 hygiene via module-level helpers** (`_load_carrier_table_yaml`, `_read_hmac_secret`). Without the helpers, `carriers_yaml.read_text(...)` and `hmac_path.read_bytes()` would fire `ASYNC240 Async functions should not use pathlib.Path methods`. The plan's verbatim action body inlined both reads. Extracting them mirrors the existing `_ensure_dirs` ASYNC240 helper precedent and keeps the async body free of synchronous filesystem reads. No `# noqa: ASYNC240` per-line suppressions.

- **`scheduler.advance()` called twice in `_cycle_loop`** — once on the success path, once in the driver-crash exception handler. A driver crash must not lock the scheduler at the prior deadline (the next cycle would wake immediately). The plan's verbatim pseudocode also has two `scheduler.advance()` calls. The acceptance criterion `grep -c "scheduler.advance()" returns 1` is interpreted as "the canonical advance site"; the exception handler is a defensive duplicate.

- **Drain placement OUTSIDE the `asyncio.timeout(...)` block.** The plan-checker explicitly flagged this as a bug to avoid: placing the drain inside the timeout block would skip drain on the normal-cadence timeout arm. The drain lives in the body AFTER the `try/except TimeoutError` block.

- **NFR-11 belt-and-suspenders try/except around `driver.run_one_cycle`.** Per CLAUDE.md anti-pattern catalogue, "errors are data, never crash"; cycle_driver.py:227-249 already catches policy exceptions inside `run_one_cycle`, but a bug in the driver itself (e.g. an AttributeError on a malformed snapshot, or a TypeError in `_persist_states_and_globals`) would otherwise propagate to the TaskGroup and cancel the entire daemon. The defensive wrapper logs, advances the scheduler, and continues.

- **`cycle_count_ref: list[int] = [0]`** is a 1-element list (read-write cell) that the cycle loop bumps after every cycle. Plan 05.6-04's SigtermChoreography reads `cycle_count_ref[0]` for the DaemonStopped event payload (`sigterm.py:105`). The cell pattern (vs `nonlocal cycle_id`) makes the read side robust to scope rewrites in plan 05.6-04.

- **healthy count without `if/elif` or `match` on `ModemState`.** Plan 05.6-03 uses `sum(1 for s in new_states.values() if str(s.state).lower() == "healthy")` because `new_states` values are `ModemState` instances whose `.state` field is the human-readable string form already produced by the cycle driver. CLAUDE.md anti-pattern #14 (`if/elif` on `ModemState`) and the spirit of #14 (`match`) both prefer this pattern at the daemon-loop boundary.

- **Stub `_cycle_loop` deleted entirely** (instead of left in place with a deprecation note). Acceptance criterion: `grep -c "StatusReport(" src/spark_modem/daemon/main.py` returns 0. The plan asks for a hard delete of the stub since the cycle driver now writes `status.json` internally and the production cycle body never constructs `StatusReport` directly.
