# Phase 05.6 — production-main-loop (SPEC)

## What this phase delivers (in one sentence)

`spark-modem-watchdog.service` on a bench Jetson reaches `Active: active
(running)` with `sd_notify READY=1` and stays there indefinitely, with
the full production TaskGroup wired (4 event producers + cycle driver
+ status writer + sd_notify lifecycle + signal handlers) inside
`acquire_pid_lock`.

## Why this is its own phase

The Phase 05.x hotfix chain (05.1 - 05.5) closed the deploy plumbing —
.deb packaging, daemon settings source, libqmi/firmware parsers, and
qmi-proxy contention retry. After all five hotfixes, the daemon
successfully:

- installs cleanly via `apt install`
- passes both `ExecStartPre` gates (postinst smoke + `ctl config-check`)
- runs `_ensure_dirs` against the production paths without EROFS
- completes all three Phase 5 X-03 preflight probes
- passes the known-fleet allow-list check (with operator-supplied fixture)
- acquires the `/run/spark-modem-watchdog/lock` PID file
- **returns 0 immediately** because the TaskGroup body is a documented
  placeholder

This is not a hotfix — it is the deferred body of Plan 03-09 that was
marked "completed approved-with-deferral" without the production-path
integration. The lifecycle scaffold (preflight, PID lock, sd_notify
wrapper, sigterm/sighup handlers, cycle driver, cycle scheduler) all
exist as modules; they were never wired together for production use.
The `--laptop` mode (`_laptop_main`) has analogous wiring against
fixtures, which is what Phase 4 HIL tests exercise.

## Bug surface (the failure mode)

`src/spark_modem/daemon/main.py` lines 261-306, inside
`acquire_pid_lock`:

```python
# Step 8 + 9 + 10 (TaskGroup + first cycle + READY + cadence)
# are wired by Plan 03-09. The placeholder below documents
# the cycle-loop body order so the WATCHDOG-cycle-end
# invariant is auditable:
#
#     async with asyncio.TaskGroup() as tg:
#         tg.create_task(restart_on_crash("udev_producer", ...))
#         ...
#
# [...documentation block continues...]
#
_ = restart_on_crash  # keep the import live for Plan 03-09
_ = signal  # keep the import live for the eventual loop.add_signal_handler calls
_ = UdevInventory  # production-path InventorySource (Plan 03-09)
_ = ZaoLogInotifyTailer  # production-path ZaoLogTailer (Plan 03-09)
return 0
```

systemd's `Type=notify` correctly reports `Result: 'protocol'` when the
process exits before `READY=1` arrives.

## In-scope (must change)

1. **Replace the placeholder with real TaskGroup wiring** inside the
   `with acquire_pid_lock(...) as ...:` block. Topology per the
   placeholder comment:
   - `udev_producer` (`pyudev.Monitor` reader → `WakeSignal` queue,
     producing `UsbAdded` / `UsbRemoved` / `SimSwapped`)
   - `rtnetlink_producer` (`pyroute2.AsyncIPRoute` → link-state events)
   - `asyncinotify_producer` (Zao log tailer + events.jsonl reopen on
     logrotate, both modes)
   - `kmsg_producer` (non-blocking `/dev/kmsg` reader → USB overcurrent
     / thermal / "device not accepting address" events)
   - `_cycle_loop` (the daemon's heartbeat — waits on `WakeSignal` OR
     poll deadline, runs one cycle via `CycleDriver`, writes
     `status.json`, fires `sd.watchdog_kick()` AT CYCLE END,
     `sd.status(...)` line)
   - `_sigterm_watcher` (sets `shutdown_event` on SIGTERM)
   - `_sighup_watcher` (sets `sighup_event` on SIGHUP)

   Each producer wrapped in `restart_on_crash(name, factory)` (already
   exists from Plan 03-06).

2. **`sd_notify` lifecycle**:
   - `loop.add_signal_handler(SIGTERM, shutdown_event.set)` and
     same for SIGHUP — installed BEFORE the TaskGroup is created
     (Pattern 7). NEVER `signal.signal()` (CLAUDE.md anti-pattern).
   - `sd.ready()` fires AFTER the first cycle's `status.json` is
     written — proves the daemon is actually doing work, not just
     up.
   - `sd.watchdog_kick()` fires AT CYCLE END (PITFALLS §4.1 — NOT
     mid-cycle).
   - `sd.status(f"cycle={result.cycle_id} ...")` once per cycle so
     `systemctl status` shows a human-readable line.

3. **Production subsystem construction** with real implementations:
   - `UdevInventory` replaces `_InventoryFromFile`.
   - `ZaoLogInotifyTailer` replaces `_NoZaoTailer`.
   - `StateStore` constructed against `settings.state_root` (the actual
     /var/lib/ path), not the laptop sandbox override.
   - `EventLogWriter` against `settings.events_log_path` (the actual
     /var/log/ path).
   - `MetricRegistry` + Prometheus UDS exporter at
     `settings.metrics_socket_path` (`/run/spark-modem-watchdog/metrics.sock`).
   - `WebhookPoster` with the real HMAC secret loaded via
     `Settings.resolve_hmac_secret_path()` (the L-02 fallback from
     Phase 05.2).
   - `CarrierTable` loaded from `/etc/spark-modem-watchdog/conf.d/`
     (per Phase 1 plan 01-06 day-one carrier YAML).
   - `CycleScheduler` with production interval (probably 30s; revisit).

4. **Graceful shutdown path** (FR-53):
   - SIGTERM → `shutdown_event.set()` → TaskGroup cancels →
     state-store flush → pre-exit best-effort webhook → `sd.status()`
     final line → exit code 0.
   - Whole sequence must complete in ≤5 seconds (Phase 3 SC #3).

5. **SIGHUP transactional reload** (FR-53, FR-54-runtime):
   - SIGHUP → `sighup_event.set()` → data-only fields (carrier table,
     thresholds, webhook URL) reloaded atomically.
   - Topology-affecting fields trigger a `logger.warning("restart required")`
     and are NOT applied.

## Out-of-scope (deliberately)

- The `--laptop` path in `_laptop_main`. It works as-is and the Phase 4
  HIL suite depends on it; leave alone.
- Refactoring `CycleDriver` or `CycleScheduler` internals. Both modules
  exist and are tested; just call them.
- New verbs in `ctl`. The existing config-check / capture-fleet-fixture /
  reset / status / explain / support-bundle / etc. surface is enough.
- Zao SDK detection improvements (the `unknown` sentinel propagating
  cleanly is fine for now; if a real fielded box turns out to need a
  concrete SDK version, that's a future phase).
- The firmware-family-mismatch concern (SWI9X50C on bench vs SWI9X30C
  in CLAUDE.md). Phase 6 / spec-doc concern.
- Replacing `qmi-proxy` reliance or designing a CID-fairness scheduler.
  Phase 05.5 retry is enough for preflight; if cycle-time diag probes
  hit the same contention, add retry at the `QmiWrapper` layer in a
  future phase.

## Acceptance criteria

1. On a bench Jetson with the operator-provisioned HMAC secret + a
   captured known-fleet entry, `systemctl start spark-modem-watchdog`
   reaches `Active: active (running)` with `sd_notify READY=1` within
   60 seconds (NFR-13). All four ExecStartPre gates remain green.
2. `systemctl status` shows a `Status:` line per cycle (sd.status
   updates).
3. After ≥10 minutes of uptime: zero ERROR/CRITICAL lines in the
   journal; `status.json` updates every cycle; `metrics.sock` returns
   valid Prometheus text via curl --unix-socket.
4. `systemctl stop spark-modem-watchdog` returns 0 within 5 seconds
   (FR-53 / Phase 3 SC #3); a `daemon_stopped` event with
   `reason: sigterm` lands in `events.jsonl` before exit.
5. `systemctl reload spark-modem-watchdog` succeeds; data-only fields
   (carrier table, webhook URL) reload without restarting; topology
   fields generate a "restart required" warning instead of misapplying.
6. Replay-mode regression: `pytest tests/` reports the same pass
   count as before this phase (no fixture-driven regressions; this
   work is additive to production-path code).
7. Integration test: spin up the production main inside a controlled
   environment (TestContainers? unittest with mocks?) and assert the
   `READY=1` notification fires. Exact test design is part of the
   plan; the success criterion is "a deterministic test catches a
   regression where someone re-removes the TaskGroup wiring."

## Suggested plan breakdown

The work splits cleanly into 3-5 plans. Suggested:

- **05.6-01** — TaskGroup skeleton + sd_notify lifecycle + signal
  handlers. Concrete: replace the placeholder with an empty
  TaskGroup that just creates the sigterm/sighup watchers + a single
  `_cycle_loop` task that does nothing useful but writes a stub
  status.json and fires `sd.ready()` after the first iteration. Acceptance:
  bench Jetson reaches `active (running)` even though no producers
  are wired. Establishes the spine.
- **05.6-02** — Wire the four producers (udev, rtnetlink,
  asyncinotify, kmsg). Each `restart_on_crash`-wrapped. Acceptance:
  status.json shows inventory updates on USB add/remove and link
  state changes; events.jsonl writes a kmsg-classified entry on a
  synthetic `dmesg` event.
- **05.6-03** — Wire production `_cycle_loop` body (the real cycle):
  `CycleDriver.run_one_cycle` → `status_reporter.write_status_json`
  → `sd.watchdog_kick()` → `sd.status(...)`. Acceptance: every cycle
  ends with a fresh status.json on disk; `actions_total` metric
  increments observed via Prometheus scrape.
- **05.6-04** — SIGTERM graceful shutdown + SIGHUP transactional
  reload (FR-53). Acceptance: `systemctl stop` returns ≤5s with
  `daemon_stopped` event written; `systemctl reload` rotates the
  carrier table without restarting.
- **05.6-05** — Integration test for the production main loop
  (deterministic; mocks systemd notify + subsystems where needed).
  Acceptance: a regression where someone reintroduces the placeholder
  return-0 is caught by `pytest` without requiring a bench Jetson.

(The 5-plan split is a suggestion; spawn `/gsd-plan-phase 05.6` to
produce the authoritative PLAN.md set after a discuss-phase pass.)

## Risk register

| Risk | Why it's a risk | Mitigation |
|------|-----------------|------------|
| Same qmi-proxy CID contention as Phase 05.5 affects cycle-time diag probes | The Zao process never stops hammering the proxy; every cycle's `qmicli` invocation can race | Add retry at `QmiWrapper` layer (similar to Phase 05.5 pattern) — but defer to a follow-up; first see whether the race actually surfaces in cycle-time logs |
| sd_notify socket setup requires `$NOTIFY_SOCKET` env var | Daemon must inherit it from systemd; missing env → silent no-op | Validate `$NOTIFY_SOCKET` is set early; log a warning if not and continue (running outside systemd is a valid dev-mode scenario) |
| Producer integration race during shutdown — TaskGroup cancellation may leave partial writes | asyncio TaskGroup propagates CancelledError; subsystems must handle it cleanly | Each subsystem already documents its shutdown contract; verify per-subsystem in plan 05.6-04 |
| Bench Jetson modems are SWI9X50C, not SWI9X30C (CLAUDE.md spec) | Some QMI parsers may have been authored against EM7421-specific output shapes | Catch any new parser mismatches the same way as 05.4: capture fixture + broaden parser; track in a future phase if it surfaces |
