**Gathered:** 2026-05-13

<domain>
## Phase Boundary

Phase 05.6 replaces the documented placeholder in
`src/spark_modem/daemon/main.py:261-306` (inside `_production_main`'s
`acquire_pid_lock` with-block) with the real production TaskGroup
wiring that Plan 03-09 was supposed to land — 4 event producers
(udev / rtnetlink / asyncinotify / kmsg) wrapped in
`restart_on_crash`, the cycle-driver loop, `sd_notify` lifecycle
(READY=1 / STATUS / WATCHDOG=1) and `loop.add_signal_handler` SIGTERM/
SIGHUP installations — so that `systemctl start spark-modem-watchdog`
on a bench Jetson reaches `Active: active (running)` and stays there.

The lifecycle scaffold modules (`daemon/lifecycle.py`, `daemon/sigterm.py`,
`daemon/sighup.py`, `daemon/preflight.py`, `daemon/cycle_driver.py`,
`daemon/cycle_scheduler.py`, `event_sources/*_producer.py`,
`event_sources/supervisor.py`, `inventory/udev.py`,
`zao_log/inotify_tailer.py`) all exist, are tested, and are imported
in `main.py` as keep-alive `_ = ...` references. The work is
genuinely "assemble," not "build."

**Carried forward from Phase 3 (locked, do not re-discuss):**

- Pattern E-01: `asyncio.TaskGroup` with each of the 4 producers
  wrapped in `restart_on_crash(name, factory)`. Cycle driver is the
  5th task; SIGTERM + SIGHUP watchers are the 6th + 7th.
- Pattern E-02: `event_queue` carries opaque `WakeSignal` enums; state
  derives from re-observation, not from event payloads.
- Pattern L-01: READY=1 fires only AFTER cycle 0 completes (real work
  done; PITFALLS §4.1). WATCHDOG=1 kicks at cycle-END after
  `status.json` is written, NOT cycle-start. STATUS= once per cycle.
- Pattern L-02: 8-step `SigtermChoreography` (already built in
  `daemon/sigterm.py`) is the SIGTERM handler. 5 s total budget.
- Pattern L-03: `SighupSwapper` (already built in `daemon/sighup.py`)
  is the SIGHUP handler. RELOAD_DATA only; RELOAD_RESTART fields are
  refused with a `restart_required` log line.
- Pattern L-04: clean-shutdown marker IO is in `daemon/lifecycle.py`
  (`write_clean_shutdown_marker` / `classify_prior_run`).
- Pattern L-05 startup order — already wired through step 7. Phase
  05.6 lands steps 8–10 (TaskGroup + first cycle + READY=1 + cadence).
- `loop.add_signal_handler(SIGTERM/SIGHUP, event.set)` installed
  BEFORE the TaskGroup is created. NEVER `signal.signal()` from
  asyncio (CLAUDE.md anti-pattern).
- `Settings()` direct construction (Phase 05.2 fix). NEVER
  `build_default_settings()` in `_production_main` — that's the CLI
  laptop-sandbox factory.
- `Settings.resolve_hmac_secret_path()` is the HMAC secret loader
  (Phase 05.2 L-02 LoadCredential→file fallback). Already implemented.
- 3-layer locks per ADR-0012 — PID lock acquired BEFORE READY=1, held
  for the entire daemon lifetime.
- `RuntimeDirectoryPreserve=yes` is load-bearing (PITFALLS §4.4) —
  the unit file already ships this; do not touch.
- The `--laptop` path in `_laptop_main` is OUT OF SCOPE — Phase 4 HIL
  tests depend on it, leave alone.

<decisions>
## Implementation Decisions

### C. Cycle cadence & warmup (SPEC §1, §2; NFR-1, NFR-13)

- **C-01: Production cycle interval = 60 seconds.** Halves the
  cycle-time qmi-proxy interaction rate vs the 30 s laptop fixture
  cadence (PITFALLS §4.5 / Phase 05.5 contention background) while
  keeping M2 MTTR targets reachable on first-bad-cycle. M2 60 s-SIM
  MTTR is "median," not "always under," so a 60 s cycle median ≤60 s
  MTTR for first-bad-cycle SIM faults is still defensible.
- **C-02: `cycle_interval_seconds` is a Settings field (RELOAD_DATA).**
  Add to `src/spark_modem/config/settings.py` mirroring
  `ladder_min_interval_seconds` / `multi_modem_threshold_fraction`:
  `Field(default=60.0, ge=1.0, json_schema_extra=RELOAD_DATA,
   description="Production cycle cadence; SIGHUP can retune live")`.
  Shadow / canary phases (Phase 5/6) can retune per cohort without
  daemon restart.
- **C-03: WATCHDOG ratio honored — bump WatchdogSec 90 → 180 s.**
  PITFALLS §4.1 prescribes 3× cycle interval. The Phase 3 systemd
  unit ships WatchdogSec=90s (matched the 30 s laptop cadence). With
  C-01 at 60 s, the unit file edit MUST land or systemd will
  watchdog-restart the daemon mid-cycle. Single edit in
  `debian/spark-modem-watchdog.service` alongside the Settings field.
  This is a downstream consequence of C-01, not a separate decision.
- **C-04: First-cycle warmup — strict READY + per-modem STATUS
  progress.** READY=1 fires only after cycle 0 completes (Phase 3
  L-01 / PITFALLS §4.1 "real work done" — non-negotiable). During
  cycle 0, the cycle driver fires interim STATUS=
  `"warmup probing modem {k}/{N} cdc-wdm={path}"` per modem probed
  so the operator sees forward progress. If cycle 0 exceeds the
  NFR-13 60 s budget under qmi-proxy contention, systemd's
  Type=notify protocol correctly fails the start — that's the right
  signal (preflight passed but cycle is broken). Phase 05.6 does NOT
  add retry to mask this — see Q-01.
- **C-05: STATUS= cadence post-warmup — per-cycle terse.** One
  STATUS= update per cycle, fired alongside `sd.watchdog_kick()` at
  cycle-end. Format: `cycle={N} healthy={K}/4 actions={M} drift={X}s`
  (Phase 3 L-01 / SPEC §2 verbatim). No per-action STATUS to keep
  sd_notify socket traffic to one packet per cycle.

### T. Integration-test strategy (SPEC §7)

- **T-01: In-process `await main(['--skip-preflight'])` with fake
  sd_notify.** New test file `tests/integration/test_production_main.py`.
  Monkeypatch `daemon.lifecycle.SdNotifyLifecycle` (or set
  `NOTIFY_SOCKET` to an abstract Unix datagram socket the test
  owns + recv on it) so READY=1 is observable. Existing
  `tests/integration/test_lifecycle.py` composes substrates
  directly via CycleDriver + SigtermChoreography → it BYPASSES
  `_production_main` entirely, so a return-0 regression on the
  placeholder slips through. T-01 closes that gap by going through
  the real main entrypoint.
- **T-02: All 4 producer injections via Fake* from `tests/fakes/`.**
  `FakeUdevMonitor`, `FakeAsyncIPRoute`, `FakeAsyncinotify`,
  `FakeKmsgReader` already exist (Phase 3). Real-Linux producer
  binding behavior is covered by their own Plan 03-02..05 tests.
  This test pins the WIRING into `_production_main`, not the
  producers' own contracts. Inject via factory hooks passed to
  `_production_main` — the planner picks the DI shape (env-overrides,
  module-level monkeypatching, or explicit factories arg). The
  laptop path's `_LAPTOP_INVENTORY_PATH` precedent + the
  cycle_driver_task / producer_tasks parameters in
  `SigtermChoreography.__init__` show the pattern.
- **T-03: Test placement — new file
  `tests/integration/test_production_main.py`** with module-level
  `pytestmark = [linux_only, asyncio]` mirroring
  `test_lifecycle.py:63-70` conventions. Selectable via
  `pytest -m linux_only`; runs on the aarch64 self-hosted CI runner
  + Linux dev hosts; skipped on Windows dev hosts cleanly.
- **T-04: Assertions — READY=1 + status.json shape + cycle metric
  incremented.** Specifically: (1) `FakeSdNotify.ready_calls` has
  exactly 1 entry within budget; (2) `status.json` exists at
  `settings.state_root/status.json` and contains 4 modems +
  `summary.expected_modems == 4`; (3) at least one of
  `actions_total` / `cycle_duration_seconds` recorded in the
  injected `MetricRegistry`. This catches both the wiring (TaskGroup
  exists and READY=1 fires from inside it) AND cycle execution
  (CycleDriver actually invoked). Matches `test_sc1_boot_to_ready`'s
  assertion shape but routed through `main()`.
- **T-05: SC #3 (SIGTERM 5 s) + SC #4 (SIGHUP) coverage stays in
  `test_lifecycle.py`.** Don't double-cover; `test_sc3_sigterm_5s`
  and `test_sc4_ctl_serialization` already exercise the
  choreography modules directly. T-01..T-04 is a single test of the
  main-loop WIRING regression, not a full SC reverification.

### Q. qmi-proxy retry at cycle time (SPEC risk register row 1)

- **Q-01: Defer cycle-time retry per SPEC.** Phase 05.6 ships
  `QmiWrapper` unchanged. Cycle-time qmicli failures surface as
  `Issue(category=qmi, detail=unexpected_output)` in events.jsonl
  via the existing per-probe error path. Rationale: the burst-rate
  qmi-proxy contention surfaced at preflight (Phase 05.5) might NOT
  manifest the same way at cycle cadence (60 s + per-probe
  interleaving with Zao traffic). Ship the loop, observe in shadow,
  then iterate. Pre-adding retry at the wrapper layer would hide
  the diagnostic signal that informs the next phase's design.
- **Q-02: Follow-up phase opens if cycle-time `qmicli` failure rate
  ≥2 per modem per hour fleet-aggregate.** Concrete operability
  gate. Phase 5 (bench/field shadow) and Phase 6 (canary) already
  track per-cycle error rates; a fleet-aggregate rate above the gate
  fires "open a phase to add retry at QmiWrapper layer using the
  Phase 05.5 3×, 0.5 s backoff pattern." Below the gate, the
  existing Diag classification handles transient failures.

### P. Plan slicing

- **P-01: Accept SPEC's 5-plan suggestion verbatim.**
  - `05.6-01` — TaskGroup skeleton + sd_notify lifecycle + signal
    handlers (`loop.add_signal_handler` for SIGTERM/SIGHUP) + the
    `cycle_interval_seconds` Settings field + WatchdogSec=180 unit
    file edit. Empty TaskGroup that creates only the signal watchers
    + a single `_cycle_loop` that writes a stub status.json and
    fires `sd.ready()` after the first iteration.
  - `05.6-02` — Wire the 4 producers (udev, rtnetlink,
    asyncinotify, kmsg) each `restart_on_crash`-wrapped.
  - `05.6-03` — Wire production `_cycle_loop` body:
    `CycleDriver.run_one_cycle` → `status_reporter.write_status_json`
    → `sd.watchdog_kick()` → `sd.status(...)`.
  - `05.6-04` — Wire `SigtermChoreography` (from `shutdown_event`) +
    `SighupSwapper` (from `sighup_event`) into the running daemon.
  - `05.6-05` — Integration test `tests/integration/test_production_main.py`
    per T-01..T-04.
- **P-02: Bench-PASS gating after 05.6-01 and after 05.6-05.**
  After 05.6-01: bench Jetson reaches `Active: active (running)`
  with empty cycle loop — first integration milestone (validates
  the spine is alive end-to-end). After 05.6-05: full SC #1–#7
  bench verification per SPEC's acceptance criteria. The middle
  plans (02, 03, 04) land back-to-back without manual checkpoints.
- **P-03: Settings.cycle_interval_seconds field + WatchdogSec=180 s
  edit land in 05.6-01.** Single unit-file edit at phase start
  (mirrors Phase 3 U-01 "one edit, no mid-rollout edits" pattern).
  The skeleton plan is the natural home — later plans consume the
  field, no one edits the unit file twice.

### Claude's Discretion

- **`_production_main` dependency-injection shape for the
  integration test.** Whether T-02 fakes are injected via explicit
  factory parameters (e.g. `_production_main(args, *, inventory_factory=...,
  zao_factory=...)`), via monkeypatching at the module level, or via
  environment-driven config — planner picks the cleanest seam.
  Constraint: the seam must NOT change the production code path's
  default behavior (no `if test_mode:` branches in production).
- **STATUS= line text format during cycle 0 warmup.** C-04 sketches
  `warmup probing modem {k}/{N} cdc-wdm={path}` but the exact text,
  per-modem-or-aggregate, and update frequency (per probe, per
  modem, per phase of cycle) is the planner's call. Keep it terse
  enough that `systemctl status` output stays readable.
- **`MetricRegistry` + Prometheus UDS start order vs cycle 0.**
  SPEC says "wire it." Whether the UDS server starts BEFORE cycle 0
  (operator can scrape during warmup; risk: scraping a registry
  with no observations yet returns empty metric responses) or AFTER
  cycle 0 (registry has at least one observation; risk: scrape
  during warmup fails with ECONNREFUSED). Planner picks; both are
  defensible. Lean toward before — the UDS server is cheap and the
  scrape protocol tolerates empty registries.
- **`CarrierTable` load path.** SPEC says "loaded from
  `/etc/spark-modem-watchdog/conf.d/`"; existing Phase 1 carrier
  YAML is shipped there. The loader call is straightforward;
  planner picks the helper (existing `wire/carriers.py` or new).
- **`WebhookPoster` HMAC secret read path.** `Settings.resolve_hmac_secret_path()`
  exists from Phase 05.2 with the LoadCredential→file fallback.
  Just call it; no decision needed.
- **`shutdown_event` / `sighup_event` placement** — module-scope vs
  passed-as-arg vs `daemon/main.py`-local. Planner picks; pattern
  is `asyncio.Event()` created inside `_production_main` body
  before `loop.add_signal_handler` calls.
- **Test seam for `FakeSdNotify`.** Whether to monkeypatch the
  `SdNotifyLifecycle` class import in `daemon.main`, or set
  `NOTIFY_SOCKET` and recv on a real abstract Unix datagram socket
  the test owns. Lean toward monkeypatch for simplicity; abstract
  socket is more faithful but ties the test to Linux (the
  `linux_only` marker covers that anyway).

</decisions>

<specifics>
## Specific Ideas

- **60 s cycle is a deliberate compromise.** Halves qmi-proxy
  interaction rate vs 30 s while keeping M2 first-bad-cycle MTTR
  targets reachable. Operators tuning shadow/canary can SIGHUP-retune
  via the RELOAD_DATA field.
- **WatchdogSec=180 is a consequence, not a separate choice.**
  PITFALLS §4.1 prescribes 3× cycle interval; the single unit-file
  edit lands in 05.6-01 alongside the Settings field so the cluster
  is one logical change.
- **Strict + STATUS progress preserves "READY = real work done."**
  PITFALLS §4.1's load-bearing principle: a daemon that reports
  Active before cycle 0 finished proven work is lying to systemd.
  Interim STATUS= updates give the operator forward-progress signal
  during warmup without compromising the principle.
- **If cycle 0 exceeds 60 s, the daemon fails systemd-start.** That
  IS the correct signal — preflight passed, cycle is broken. Q-01
  defers cycle-time retry precisely because masking this failure
  would hide the diagnostic that informs Q-02's follow-up phase.
- **In-process `main()` test catches the exact regression SPEC §7
  demands.** Going through the real entrypoint (vs Phase 3's
  direct-composition pattern) means a return-0 placeholder would
  fail the assertion `len(sd.ready_calls) == 1` because the
  TaskGroup never runs. The direct-composition tests stay valuable
  for their original SC coverage; this one specifically pins the
  main-loop WIRING.
- **All-Fake* producer injection isolates the test from
  pyudev/pyroute2/asyncinotify/kmsg binding.** The producers'
  own Phase 3 plan tests cover those bindings; T-01..T-04 pins
  what `_production_main` does WITH them.
- **Deferring cycle-time qmi-proxy retry is the SPEC's explicit
  call.** Q-02 turns that into a concrete operability trigger
  (≥2 failures/modem/hour fleet-aggregate) so Phase 5/6 has a clean
  "open a phase" signal rather than ambiguous handwaving. The
  observability ALREADY exists — `actions_total` + `cycle_duration_seconds`
  + the Issue events in events.jsonl give the signal.
- **5-plan slicing is the SPEC's suggestion, accepted verbatim.**
  Each plan is independently bench-PASSable. P-02 limits bench
  Jetson human-verify checkpoints to two — after 05.6-01 (spine
  alive) and after 05.6-05 (full SC verification) — so the middle
  plans land back-to-back without manual gating. Phase 3's 9 plans
  for analogous wiring + scope is precedent for granular bench
  checkpoints not being needed at every plan boundary.
- **The unit-file edit at 05.6-01 mirrors Phase 3 U-01's "one
  edit, no mid-rollout edits" pattern.** Single source of truth
  across the phase.
