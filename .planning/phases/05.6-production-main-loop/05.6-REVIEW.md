## Summary

Phase 05.6 wires the production `_production_main` TaskGroup spine: 4 supervised producers, the real cycle body with `CycleDriver`/`StateStore`/`MetricRegistry`/`WebhookPoster`/Prometheus UDS, `SigtermChoreography` with a 5 s deadline, and `SighupSwapper` for live `RELOAD_DATA` swaps. The `Settings.cycle_interval_seconds` field is correctly marked `RELOAD_DATA`, `WatchdogSec=180s` is bumped to 3 x the 60 s production cycle, and the new `test_production_main.py` exercises the whole TaskGroup in-process.

Overall the wiring is sound and respects every critical invariant scanned for: monotonic-only arithmetic (ADR-0007), no subprocess/httpx in policy, no inbound IPC in the unit file, integer-encoded metrics, list-form argv (only Python-side here), atomic per-cycle status writes inside `CycleDriver`. The five-state machine is composed from `result.cycle_result.new_states` correctly.

Two `Warning`-grade issues were found:

1. **`sd.ready()` can be silently skipped forever if cycle 0 crashes** (`main.py:806`). The `if cycle_id == 0` gate runs only on the success path; the crash branch at line 778 increments `cycle_id` to 1, so a recovery on cycle 1 will never fire READY. systemd masks this with `TimeoutStartSec` (the daemon gets restarted), but the latent intent ("fire READY on the first SUCCESSFUL cycle") is not what the code expresses.
2. **`webhook_poster._dns_cache` private-attribute access** (`main.py:564`) crosses the encapsulation boundary; a refactor of `WebhookPoster` could silently break SIGHUP DNS refresh.

The remaining six items are `Info` (stale docstrings from a plan-to-plan handoff, an in-function class definition, dead `del scheduler`, `__import__` curio, laptop-path webhook poster never drained, and a slightly weak test-timeout failure message).

No security findings; no critical findings.

## Warnings

### WR-01: `sd.ready()` skipped on cycle-0 crash, never fires on subsequent recovery

**File:** `src/spark_modem/daemon/main.py:806`
**Issue:**
The cycle loop fires `sd.ready()` only on the `cycle_id == 0` success branch. If cycle 0 crashes (lines 771-781) `cycle_id` is bumped to 1 in the `except` arm before `continue`. When cycle 1 then succeeds, the predicate `if cycle_id == 0` is False and READY is never sent. The daemon then runs forever (or until `TimeoutStartSec` expires and systemd kills it).

The semantic intent stated in the comment is "READY=1 only after cycle 0 (PITFALLS section 4.1)" — but the operational meaning is "fire READY when the first cycle actually succeeds." Today the code conflates "cycle 0" with "first successful cycle" and they diverge on the crash path.

This is masked in practice because `systemd Type=notify` will kill a daemon that doesn't notify READY within `TimeoutStartSec` (default 90 s), so a chronic-crash failure mode looks like a `Restart=on-failure` loop. But the latent bug means a *transient* cycle-0 crash followed by cycle-1 success leaves the daemon in a permanently "starting" state.

**Fix:** Track READY separately from `cycle_id`. Example:

```python
ready_fired = False
# ...
async def _cycle_loop() -> None:
    nonlocal ready_fired
    cycle_id = 0
    while not shutdown_event.is_set():
        # ... wait ...
        try:
            result = await driver.run_one_cycle(cycle_id=cycle_id)
        except Exception:
            logger.exception("cycle %d crashed in driver; continuing", cycle_id)
            cycle_id += 1
            cycle_count_ref[0] = cycle_id
            scheduler.advance()
            continue

        # ... status / watchdog ...

        if not ready_fired:
            sd.ready(f"first cycle ok healthy={healthy_count}/4")
            ready_fired = True

        cycle_id += 1
        scheduler.advance()
```

`nonlocal` is needed because `_cycle_loop` is a closure; alternatively use the same single-cell list pattern already used for `cycle_count_ref`.

### WR-02: Private-attribute access into `WebhookPoster._dns_cache` from `_production_main`

**File:** `src/spark_modem/daemon/main.py:564`
**Issue:**
`SighupSwapper` is constructed with `dns_cache=webhook_poster._dns_cache`. The leading underscore signals "private to `WebhookPoster`"; this implicit contract is not enforced and could be silently broken by a future refactor of the poster (rename, lazy init, replacement with a typed factory) without test coverage flagging the breakage.

This crosses an encapsulation boundary in a load-bearing way: W-02's DNS force-refresh on `webhook_url` change is one of the SIGHUP swap's documented side effects (sighup.py line 118-126). Losing it silently degrades a failure-mode that was explicitly designed for.

**Fix:** Expose `dns_cache` as a public property on `WebhookPoster` (read-only), or accept a `DnsCacheProto` in `WebhookPoster.__init__` and pass the same instance into `SighupSwapper`:

```python
# In WebhookPoster:
@property
def dns_cache(self) -> DnsCacheProto:
    return self._dns_cache

# In _production_main:
sighup_swapper = SighupSwapper(
    settings_ref=settings_ref,
    settings_factory=Settings,
    dns_cache=webhook_poster.dns_cache,
)
```

The latter (pass-in) is more testable; the former (property) is the minimum-diff fix.

### IN-02: `_AsyncioSleeper` and `_SettingsRef` defined inside `_production_main`

**File:** `src/spark_modem/daemon/main.py:407-413, 473-485`
**Issue:**
Two small classes are defined inside the async function body. Each call to `_production_main` re-creates the class objects, and they cannot be unit-tested or type-checked in isolation. They are stateless / single-cell helpers — natural module-level fits.

**Fix:** Hoist both to module level:

```python
# Module level:
class _AsyncioSleeper:
    """Production Sleeper - wraps asyncio.sleep for restart_on_crash backoff."""
    async def sleep(self, delay: float) -> None:
        await asyncio.sleep(delay)


class _SettingsRef:
    """Single-cell mutable container around the current frozen Settings."""
    def __init__(self, initial: Settings) -> None:
        self._cur = initial
    def get(self) -> Settings:
        return self._cur
    def set(self, new: Settings) -> None:
        self._cur = new
```

Non-blocking; pure code-organization improvement.

### IN-06: `_drive_shutdown_after_ready` 10 s timeout failure message is non-actionable

**File:** `tests/integration/test_production_main.py:225-247`
**Issue:**
If `fake_sd.ready_calls` never gets populated within 10 s, the helper raises `SIGTERM` anyway and lets the assertion at line 269 (`len(fake_sd.ready_calls) == 1`) fail with the message "READY=1 must fire exactly once after cycle 0; got 0 calls." That doesn't tell the maintainer whether the daemon hung, the cycle crashed inside the driver, or the producer monkeypatch leaked.

**Fix:** Capture and re-surface the timeout reason. Pattern:

```python
ready_timed_out = False
async def _drive_shutdown_after_ready() -> None:
    nonlocal ready_timed_out
    deadline = asyncio.get_running_loop().time() + 10.0
    while asyncio.get_running_loop().time() < deadline:
        if fake_sd.ready_calls:
            await asyncio.sleep(0.1)
            assert (integration_run_dir / "lock").exists(), ...
            signal.raise_signal(signal.SIGTERM)
            return
        await asyncio.sleep(0.05)
    ready_timed_out = True
    signal.raise_signal(signal.SIGTERM)

# Then in the assert section:
assert not ready_timed_out, (
    "READY=1 timed out at 10s; daemon did not complete cycle 0. "
    f"watchdog_calls={fake_sd.watchdog_calls} status_calls={fake_sd.status_calls}"
)
```

This is low-priority — the existing message will still indicate failure; it just won't pinpoint the cause as quickly.
