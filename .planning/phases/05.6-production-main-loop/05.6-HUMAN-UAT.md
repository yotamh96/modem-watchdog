## Gaps

### G2 — `_UnixWSGIServer.server_bind` missing server_name/server_port assignment (Phase 05.6 deterministic regression)

- **Symptom:** Daemon crashes in `_production_main` at line 605 (`start_metrics_server`) with:
  ```
  File ".../status_reporter/prom.py", line 71, in server_bind
      self.setup_environ()
  File ".../wsgiref/simple_server.py", line 56, in setup_environ
      env['SERVER_NAME'] = self.server_name
  AttributeError: '_UnixWSGIServer' object has no attribute 'server_name'.
                  Did you mean: 'server_bind'?
  ```
- **Reproducer:** `sudo /opt/spark-modem-watchdog/python/bin/python3.12 -m spark_modem.daemon.main` (no `--skip-preflight`; preflight passed, then crash).
- **Root cause:** `_UnixWSGIServer.server_bind` (`prom.py:63-71`) calls
  `UnixStreamServer.server_bind(self)` (plain AF_UNIX socket bind, no
  attribute side effects), then `self.setup_environ()`. But
  `setup_environ()` reads `self.server_name` / `self.server_port`, which
  in the TCP path are set by `HTTPServer.server_bind` from
  `self.server_address[:2]`. AF_UNIX `server_address` is a single path
  string — `server_name` / `server_port` are never assigned.
- **Phase 05.6 regression?** YES. Phase 05.6 plan 05.6-03 wired
  `start_metrics_server` into the TaskGroup. The function existed before
  but was untested end-to-end (a Plan 03-04 placeholder). The
  `_UnixWSGIServer` class was implemented but the `server_bind` path was
  never exercised against a real socket; the plan's docstring at
  `prom.py:67-70` literally says "values are nonsense for UDS but the
  WSGI handler does not consume them" — but the values are never set.
- **Integration-test blind spot:** `tests/integration/test_production_main.py`
  monkeypatches `start_metrics_server` to `_FakePromServer()` to avoid
  binding a real AF_UNIX socket in-process. This is exactly the
  Generator-self-evaluation blind spot Anthropic's harness engineering
  research warns about (CONTEXT.md `<execution_context>` mentions this
  via `post-merge-gate.md`). The test reported PASSED while the real
  code path is broken.
- **Fix (3 lines):** In `prom.py:server_bind`, manually assign
  `self.server_name = "localhost"` and `self.server_port = 0` before
  calling `setup_environ()`. The values are unused but must exist.
- **Test fix:** Add a unit test that instantiates `_UnixWSGIServer` at a
  `tmp_path` socket, asserts no exception, asserts `server_name` /
  `server_port` are set, and unbinds. Catches the regression without
  asyncio plumbing.
- **Severity:** Critical — daemon does not run on production hardware
  once fleet-triple passes.

### G1 — Daemon's qmi-proxy child races Zao's qmi-proxy for CIDs (was: "fleet-triple retry budget")

**Hypothesis updated 2026-05-18 10:04 UTC** after the G2 hotfix unblocked
deeper diagnostics. Previous framing ("retry budget too short") was wrong.

- **Symptom:** Daemon exits 78/CONFIG ~2s after `systemctl start`,
  3 retries in a row with identical errors. Concrete qmi-proxy stderr:
  ```
  qmicli exit_code=1 stderr=
    [18 May 2026, 10:04:27] -Warning ** Cannot read from istream: connection broken
    error: couldn't open the QmiDevice: endpoint hangup
  ```
- **Triage results (2026-05-18):**
  1. ✓ All 4 modems respond to `sudo qmicli -p -d /dev/cdc-wdmN --dms-get-revision` while Zao is running (firmware `SWI9X50C_01.14.03.00`).
  2. ✓ ModemManager: `masked` + `inactive`.
  3. ✓ Zao stack active: `zao-infra-ctrl`, `zao-remote-endpoint`, plus support services. **System qmi-proxy** is `/usr/libexec/qmi-proxy` PID 2133893 (started 04:46 today, used by Zao).
  4. `lsof /dev/cdc-wdm*` shows no exclusive holders — qmi-proxy is cooperative.
  5. Every daemon `systemctl start` spawns a **fresh qmi-proxy child** under its process tree (`Killing process N (qmi-proxy)` in journal at each exit). systemd kills it when the daemon exits.
- **Root-cause hypothesis (NEEDS VERIFICATION in 05.7):**

  The daemon's qmicli spawns its OWN qmi-proxy child instead of sharing
  the system one. That fresh proxy races Zao's qmi-proxy for QMI client
  IDs from the modem firmware; the modem returns `endpoint hangup`
  because too many client allocations are outstanding.

  **CORRECTED 2026-05-18 11:00 UTC** — earlier "PrivateTmp" hypothesis
  was wrong: `debian/spark-modem-watchdog.service` line 71 explicitly
  documents "No PrivateTmp=" (drops for LoadCredential compat). So
  /tmp IS shared between daemon and Zao. The cause must be elsewhere:

  Open questions Phase 05.7 needs to answer:
  1. Where does the system qmi-proxy listen? `/tmp/qmi-proxy`?
     `/run/qmi-proxy`? Something else? (`lsof -p $(pidof qmi-proxy)`
     would tell us.)
  2. Where does qmicli's `-p` look by default? (Likely `/tmp/qmi-proxy`
     per libqmi defaults, but service-aware fork may differ.)
  3. Why does each daemon `systemctl start` spawn a new qmi-proxy
     PID (2129337, 2182303, 2182632 — fresh on every start)? Are
     they actually attached as children of qmicli, or does qmicli's
     `-p` always create a NEW proxy?
  4. Is the 04:46 system qmi-proxy at PID 2133893 actually serving
     Zao, or was it itself orphaned from an earlier daemon attempt?

  Evidence collected so far:
  - Manual `sudo qmicli -p ...` works while Zao is active.
  - Daemon's qmicli fails consistently with `endpoint hangup`.
  - The error pattern is consistent across 3 retries 1s apart.
- **Possible fixes (Phase 05.7 candidate plans):**

  1. **Share Zao's qmi-proxy.** Set `PrivateTmp=false` in the service unit
     (or `BindReadOnlyPaths=/tmp/qmi-proxy`). Daemon's qmicli finds the
     system qmi-proxy socket and reuses Zao's CID pool cooperatively.
     **Lowest blast radius, most aligned with CLAUDE.md "Zao requires
     exclusive access" — actually Zao requires exclusive USB access, not
     exclusive qmi-proxy.**
  2. **Use direct QMI (no -p flag).** Daemon talks to /dev/cdc-wdmN
     directly. Requires `BusyBox`-style CID allocation and conflicts with
     Zao's exclusive USB binding — likely NOT viable.
  3. **Stop spawning a child qmi-proxy.** Pre-start an explicit
     `qmi-proxy.service` that both Zao and daemon depend on. More invasive.

  Option 1 is the right entry point. Plan should add a unit-file edit AND
  a unit-test that audits `PrivateTmp` (extending
  `tests/integration/test_unit_file_audit.py`).
- **Phase 05.6 regression?** No — pre-existing architectural conflict
  uncovered now that the G2 prom.py crash no longer hides it. Phase 05.5's
  retry hotfix mitigates the symptom in some boots but cannot prevent the
  CID exhaustion when Zao is heavily active.
- **Blast radius:** SC #1 + cascade SC #2..#5 stay blocked until G1 is
  fixed. SC #6 + SC #7 (pytest on aarch64 CI) remain independent.
- **Recommended path:** Open Phase 05.7 "qmi-proxy sharing" via
  `/gsd-plan-phase 05.6 --gaps` — this UAT file is the gap source.
