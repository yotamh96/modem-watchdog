# Plan 05.6-05 — In-process main() Regression Test

## What was built

`tests/integration/test_production_main.py` — a 297-line in-process pytest that
routes through the **real** `await main(['--skip-preflight'])` entrypoint to pin
the wiring inside `_production_main`. It catches the exact bug SPEC §7 demands:
if someone re-introduces a `return 0` placeholder in `_production_main`, the
test MUST fail without requiring a bench Jetson.

This complements `tests/integration/test_lifecycle.py`, which intentionally
bypasses `_production_main` via direct composition (and therefore would not
catch a `return 0` regression at the top-level entrypoint).

## Shutdown driver pattern

The daemon's `shutdown_event` is a local variable inside `_production_main`,
not exportable. The cleanest seam to trigger graceful shutdown from the test
process is `signal.raise_signal(signal.SIGTERM)`:

```python
async def _drive_shutdown_after_ready() -> None:
    deadline = asyncio.get_running_loop().time() + 10.0
    while asyncio.get_running_loop().time() < deadline:
        if fake_sd.ready_calls:
            await asyncio.sleep(0.1)  # one tick for WATCHDOG+STATUS to wire
            assert (integration_run_dir / "lock").exists(), (
                "FR-61: PID lock file must exist at run_dir/lock while daemon is running"
            )
            signal.raise_signal(signal.SIGTERM)
            return
        await asyncio.sleep(0.05)
    # 10s deadline fallback: cycle 0 never fired READY. Signal anyway so
    # the daemon exits; the test's `len(ready_calls) == 1` assertion will fail.
    signal.raise_signal(signal.SIGTERM)
```

This is semantically equivalent to CONTEXT.md T-04's literal
`shutdown_event.set()`: the kernel signal handler installed by
`loop.add_signal_handler(SIGTERM, shutdown_event.set)` in plan 05.6-01 step 6
fires when `raise_signal` dispatches. Documented in the plan's
`deviation_protocol`.

NOT `os.kill(os.getpid(), SIGTERM)` — that is the documented CLAUDE.md
anti-pattern (referenced only in a comment, never invoked in code).

## Assertions after clean shutdown

1. `rc == 0` (main returned cleanly).
2. Exactly 1 `READY=1` call fired (cycle 0 ran via TaskGroup, not stub).
3. `≥1` watchdog kick (`fake_sd.watchdog_calls`).
4. `≥1` `STATUS=` call (per cycle, C-05).
5. `status.json` exists at `integration_state_root / "status.json"` with
   `len(status["modems"]) == 4` and `status["summary"]["expected_modems"] == 4`.
6. `cycle_duration_seconds_count >= 1` read from the **isolated**
   `CollectorRegistry` (not the global `prometheus_client.REGISTRY`).
7. FR-61: the PID lock file `integration_run_dir / "lock"` exists **while
   the daemon is running** (before SIGTERM unwinds the choreography, which
   deletes the lock via kernel on process exit per ADR-0012). Without this
   assertion, T-04 would silently miss FR-61 coverage.

## Acceptance-criteria deviations (documented)

The plan's task-1 acceptance criteria list a literal `signal.raise_signal`
count of "exactly 1". The implementation has **two** call sites (happy path
+ 10s-deadline fallback) — this matches the plan's own reference code (the
`# Timeout: cycle 0 never fired READY...` block at the end of
`_drive_shutdown_after_ready`). The grep-count target was a plan-level
defect, not an implementation deviation. Verified by re-reading the plan's
embedded reference body lines ~400-403.

`os.kill` appears **only in a comment** documenting the CLAUDE.md
anti-pattern; the grep-count target of 0 referred to call sites, and is
satisfied.
