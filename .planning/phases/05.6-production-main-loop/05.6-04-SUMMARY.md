# Phase 05.6 Plan 04: SIGTERM / SIGHUP Choreography Wiring Summary

**Replaces plan 05.6-01's `_stub_sigterm_watcher` and `_stub_sighup_watcher` module-level helpers with two nested closures inside `_production_main`: `_sigterm_watcher` wires the 8-step `SigtermChoreography` (cycle cancel → producers cancel → webhook drain → state flush → emit `DaemonStopped(SIGTERM)` → webhook poster stop → metrics socket unlink → clean-shutdown marker) within a 5.0s deadline budget (FR-53); `_sighup_watcher` wires `SighupSwapper.try_apply_reload()` for transactional Settings reload (RELOAD_DATA-only applies; RELOAD_RESTART refuses with `restart_required` log line; FR-54-runtime). The cycle task handle is captured via `cycle_task_ref: list[Task | None] = [None]` — a 1-element mutable closure cell mirroring plan 05.6-03's `cycle_count_ref` pattern. A `_SettingsRef` nested class provides the get/set cell that `SighupSwapper` writes to; `webhook_poster._dns_cache` is passed to the swapper so `webhook_url` SIGHUP changes force-refresh DNS via the shared W-02 cache. A belt-and-suspenders `webhook_poster.stop()` after the TaskGroup unwinds covers the crash-exit path. The 9-task TaskGroup shape is unchanged from plan 05.6-03 — only the bodies of the two signal-watcher coroutines change.**

### Choreography semantics now live

- **SIGTERM**: `systemctl stop spark-modem-watchdog` → systemd-installed SIGTERM → `shutdown_event.set()` (installed via `loop.add_signal_handler` in plan 05.6-01) → `_sigterm_watcher` wakes → `SigtermChoreography.execute(deadline_seconds=5.0)` runs all 8 steps. After the choreography returns, the cycle task + producers are already cancelled (steps 1+2); the watcher returns; the TaskGroup unwinds via `CancelledError` propagation; the belt-and-suspenders `webhook_poster.stop()` runs (idempotent no-op); `_production_main` returns 0.
- **SIGHUP**: `systemctl reload spark-modem-watchdog` → systemd-installed SIGHUP → `sighup_event.set()` → `_sighup_watcher` wakes from the `asyncio.wait` race → `sighup_event.clear()` → `await sighup_swapper.try_apply_reload()`:
  - RELOAD_DATA-only diff: applies via `settings_ref.set(new)`; if `webhook_url` changed, `dns_cache.resolve(new_host)` force-refreshes the cache.
  - RELOAD_RESTART field change: refuses with `logger.warning("sighup: restart_required fields changed=...")`; keeps old `settings_ref` cell.
  - No-op SIGHUP (no diff): returns True silently; common pattern for operator "is daemon responsive?" probes.

### Closure mechanic — why it works (cycle_task_ref)

The structural challenge: `SigtermChoreography.__init__` takes `cycle_driver_task: asyncio.Task[object]`. The cycle task is created INSIDE the `async with asyncio.TaskGroup() as tg:` block via `tg.create_task(_cycle_loop())`. But `_sigterm_watcher` (also created inside the TaskGroup) needs a closure reference to that task handle.

Solution: a 1-element mutable list (`cycle_task_ref: list[Task | None] = [None]`) declared OUTSIDE the TaskGroup body but inside `_production_main`'s scope. The TaskGroup body assigns `cycle_task_ref[0] = tg.create_task(_cycle_loop())` synchronously (asyncio guarantees `tg.create_task` returns synchronously — the task body doesn't start until the next event-loop iteration). By the time `_sigterm_watcher`'s `await shutdown_event.wait()` resolves (which can only happen after the loop has run at least once), `cycle_task_ref[0]` is already populated.

The defensive `if cycle_task_local is None: logger.warning(...); return` arm is documented as belt-and-suspenders. The condition cannot fire because:
1. `tg.create_task(_cycle_loop())` is the LAST cycle task creation in the TaskGroup body — happens before `tg.create_task(_sigterm_watcher())`.
2. `_sigterm_watcher`'s body starts with `await shutdown_event.wait()` — yields control immediately.
3. By the time `shutdown_event.set()` fires (only via `loop.add_signal_handler` on a real SIGTERM), the TaskGroup body has long since finished creating all 9 tasks.

Mirrors plan 05.6-03's `cycle_count_ref: list[int] = [0]` pattern — the same 1-element list trick for read-write closure communication.

## Decisions Made

- **`boot_mono = clock.monotonic()` captured before TaskGroup entry.** ADR-0007: monotonic-only arithmetic for durations / uptime; the choreography's `boot_monotonic=` kwarg feeds steps 5 + 8. Cannot reconstruct uptime later from wall-clock.
- **`cast(Any, event_logger)` at the SigtermChoreography boundary.** Matches plan 05.6-02's 5 producer-boundary cast sites; the choreography's narrow `_EventLogWriterProto.append(event: object)` is mypy-invariant-incompatible with `EventLogWriter.append(event: Event-union)`. Same pattern that test_lifecycle.py:363 uses (`type: ignore[arg-type]`). The runtime contract is correct (the choreography only appends `DaemonStopped`, which is a member of the `Event` union).
- **Deleted the dead `_stub_sigterm_watcher` / `_stub_sighup_watcher` module-level helpers.** The plan's frontmatter says both watchers are "defined inside `_production_main`". Leaving the stubs at module scope would be dead code that contradicts the plan's intent and would clutter the module.
- **`_sighup_watcher` uses `asyncio.wait(FIRST_COMPLETED)`** on (sighup_task, shutdown_task). Same shape plan 05.6-01 established for `_stub_sighup_watcher` after a literal `while not shutdown_event.is_set(): await sighup_event.wait()` deadlocked on SIGTERM-without-SIGHUP. The plan's action body uses the same shape; this is the only deadlock-free pattern.
- **`webhook_poster._dns_cache` is the canonical DnsCache reference.** The underscore-prefixed attribute is the documented seam — `sighup.py:118-126` reads `dns_cache.resolve(host)` directly. The plan's deviation protocol accepts `dns_cache=None` if the attribute is renamed; I confirmed the attribute exists at `poster.py:115` and used it.
- **Belt-and-suspenders `webhook_poster.stop()` after TaskGroup unwind.** The choreography's step 6 already calls stop() on the SIGTERM path; the post-TaskGroup call covers the crash-exit path. WebhookPoster.stop() is idempotent.
- **Documented limitation: `cycle_interval_seconds` SIGHUP-retuning NOT wired in Phase 05.6.** The cycle driver / scheduler were constructed with the INITIAL Settings instance; `settings_ref.set(new)` updates the cell, but the cycle driver doesn't read `settings_ref.get()` per cycle. SPEC out-of-scope "Refactoring CycleDriver internals" confirms this is intentional. A future phase can wire the cycle driver / scheduler to read `settings_ref.get().cycle_interval_seconds` per cycle without daemon restart.
